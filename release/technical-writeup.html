Release evidence

Building a trustworthy local chess tutor

The difficult part was not asking Stockfish for a score. It was preserving a coherent tutoring loop while work, failures, restarts, and user interactions happen concurrently.

One source of chess truth

The C++ board reconstructs legal positions and feeds both analysis and drills. The browser receives FEN, moves, evaluations, evidence and schedule state; it does not duplicate chess rules. Known perft positions and long make/unmake sequences protect that boundary.

Progress without starvation

Phase 2 used one shallow-first job worker. Phase 3 separates game scheduling from engine execution. Both layers maintain interactive, current-game and historical queues. Historical work is deliberately prevented from consuming the final slot. A failed Stockfish instance is replaced without stopping the pool.

Evidence before optimization

A bitboard candidate was implemented behind the same model contract and tested against the array oracle. It lost the measured paths because rebuilding bitboards costs more than the candidate saves. The honest optimization was not adopting it. In contrast, bounded caches and two isolated workers address demonstrated memory and throughput risks.

Crash semantics as product behavior

The event log acknowledges only after sync. Tests interrupt before writing, halfway through a record, before sync and after sync. They distinguish acknowledged durability from valid at-least-once replay of a complete unacknowledged record. Snapshots and indexes remain disposable.

Coaching without opaque generation

Personal-game drills remain first choice. Optional public puzzles are used only for recurring motifs and nearby rating. The original corpus setup move is applied, the presented FEN and solution are checked for legality, two fresh engine instances must agree, and close alternatives are rejected. No transformed position is shipped.
